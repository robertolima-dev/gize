# ADR-021: Authorization defaults (roles + `users` resource hardening)

- **Status:** Accepted
- **Date:** 2026-07-08
- **Deciders:** Gize core team

## Context

We are in the RC phase (0.8.x) heading to 1.0, where the **generated-code contract** gets a
SemVer promise. ADR-013 shipped authentication (Argon2id + JWT), and its security review — plus
the RC2 review in `SECURITY.md` — flagged an open **authorization** gap:

- `require_auth` checks that a request carries a **valid token**; it does **not** check roles or
  ownership. Out of the box, **any authenticated user can call any guarded write route**,
  including modifying or deleting *other* users (`PUT`/`DELETE /users/:id`).
- Generated `GET /users` and `GET /users/:id` are **public**, exposing names, emails and the
  `is_admin` flag, and enabling user enumeration.

The backlog tracks two open decisions (BACKLOG "Riscos do RC"): whether to turn authorization
into a `require_admin`/opt-in helper, and whether to change the public-read default for `users`.
Both **touch the generated surface**, so under the 0.8.0 feature freeze this is the *last window*
to decide them: RC is exactly when hardening/stabilization of the generated contract happens
(`STABILITY.md` marks the exact generated bytes as not-yet-stable), and after 1.0 either change
would be breaking.

Constraints from our philosophy: **no magic** (the developer owns readable Axum/SQLx code, no
hidden query rewriting), keep the surface **small**, ADR before code.

## Alternatives

**A. Authorization mechanism**

1. **Keep `require_auth` only + document** (status quo). Zero surface change; but ships a sharp
   footgun on the one resource where it hurts most (`users`), and the docs already warn about it
   without fixing it.
2. **Add a `require_admin` role guard** (extractor/middleware gating on `is_admin`) in the
   generated auth module, mirroring `require_auth`. Small, additive, idiomatic; makes the
   already-generated `is_admin` flag actually usable as a gate. Ownership stays the developer's
   job.
3. **Auto-inject ownership** (`WHERE owner_id = current_user`) into generated queries for
   resources with a user FK. Rewrites every generated query — magic, brittle, guesses domain
   semantics. Against ADR philosophy.
4. **Full RBAC / policy layer** (roles + permissions tables, policy DSL). Large feature; violates
   the feature freeze and the "MVP small" risk. Out of scope for 1.0.

**B. `users` resource defaults (the freeze-sensitive part)**

1. **Keep writes at `require_auth`, reads public** (status quo) — any authenticated user edits/
   deletes any user; anyone reads all emails.
2. **Harden only the `users` module**: admin-gate its management writes and stop exposing reads
   publicly, leaving generic resources untouched. Scoped surface change to one template.
3. **Change defaults for all resources** (guard all reads, admin-gate all writes). Breaks the
   deliberate "reads public, writes guarded" model globally (a blog's `GET /posts` should stay
   public); large, opinionated surface change.

## Decision

Adopt **A.2 + B.2** — a minimal role guard plus targeted hardening of the identity resource,
keeping generic resources' model intact:

1. **Ship `require_admin`** in the generated auth module (`src/auth`), alongside `require_auth`:
   an extractor/guard that resolves the authenticated user and rejects with **403** unless
   `is_admin` is true (`401` still for missing/invalid token). Plain, replaceable Axum code.

2. **Harden the generated `users` resource** (secure-by-default for the identity module only):
   - `POST/PUT/DELETE /users` (managing *other* users) default to **`require_admin`**.
   - `GET /users` (list) and `GET /users/:id` default to **`require_admin`** — no longer public;
     PII/enumeration is closed by default.
   - `register` and `login` stay **public** (unchanged); `register` still forces
     `is_admin = false` (ADR-013).
   - **Ownership / self-service** (a user reading or editing *their own* record) is **not**
     auto-generated — it is domain-specific. We document the pattern (compare the `sub` claim to
     the path id) and leave the code to the developer.

3. **Generic resources are unchanged**: writes default to `require_auth`, **reads stay public**.
   That model is deliberate (public `GET /posts`). The docs state plainly that role/ownership
   authorization on generic resources is the developer's responsibility, with `require_admin` and
   the ownership snippet as the building blocks.

4. This is gated behind auth being enabled (`features.authentication`); with auth off, behavior is
   unchanged.

## Trade-offs

- (+) Closes the two concrete holes (any-user-edits-any-user; public email/enumeration) **by
  default**, on the one resource where they are most dangerous, without a policy engine.
- (+) `require_admin` is additive and idiomatic — the developer can read it and swap it for their
  own policy; the generated `is_admin` flag finally has a use.
- (+) Keeps the "reads public, writes guarded" model for generic resources — no surprise for
  blog-style APIs; the surface change is confined to the `users` templates.
- (−) **Breaking change to the generated `users` surface** (reads now 401/403; writes now admin).
  Acceptable and intentional: it lands *now*, during RC, which is exactly when the generated
  contract may still change before the 1.0 SemVer promise. Requires updating snapshots, the e2e,
  and the reference apps (blog/shop/chat).
- (−) No automatic ownership means self-service user profiles still need developer code; mitigated
  by documenting the exact snippet. We consciously avoid query-rewriting magic.
- (−) Slightly larger auth template (one more guard) and one more status code (403) in the
  generated contract.

## Consequences

- `gize-templates` (auth + `user` module + generic routes): generate `require_admin`; wire
  `users` reads/writes to it; keep `register`/`login` public. Generic route templates gain a
  documented seam but keep current defaults.
- **Snapshots + e2e**: update golden files and the "compile → serve → call routes" checks to the
  new `users` expectations (401 without token, 403 as non-admin, 200/201 as admin; reads guarded).
  Reference apps regenerated via `gize sync`.
- **`SECURITY.md`**: update the "Authorization" and "Public read routes" sections — from
  "your responsibility / documented gap" to "`users` hardened by default; `require_admin` provided;
  ownership and generic-resource authz remain the developer's job".
- **BACKLOG**: closes the two open RC risks (`require_admin`/authz default; public reads of
  `users`); this is the surface-freezing step that should land before WS-RC5 (docs) and WS-RC6
  (external validation) so those happen against the final 1.0 surface. Target bump: **0.8.3**.
- **1.0 SemVer**: after this, the generated authorization surface is what 1.0 promises; further
  authz (RBAC, ownership helpers, configurable policies) is post-1.0 / v2.0, additive where
  possible.
- A short **security review** accompanies the implementation (correct 401-vs-403 split, admin
  guard cannot be bypassed, no PII leak in error bodies/logs), consistent with ADR-013's process.
